Quickstart

IonShield exposes a REST API. No SDK required — any HTTP client works.

1. Get current space weather

curl https://ionshield.io/api/status

2. Get a comms decision

curl "https://ionshield.io/api/v2/comms-decision?lat=38.8&lon=-77.0"

3. Get a route decision

curl -X POST https://ionshield.io/api/v2/route-decision \ -H "Content-Type: application/json" \ -d '{"waypoints":[{"lat":38.8,"lon":-77.0,"name":"Start"},{"lat":40.0,"lon":-75.0,"name":"End"}], "platform":{"asset_type":"GPS_L1","criticality":3}}'

Self-Hosting

IonShield ships as a single Python package (FastAPI + SQLite). For production, PostgreSQL is recommended.

git clone https://github.com/ionshield/ionshield-backend cd ionshield-backend pip install -r requirements.txt cp .env.example .env # edit API keys and config python -m uvicorn app.main:app --reload --port 8000 # Dashboard → http://localhost:8000/dashboard

Authentication

By default, all endpoints are open (suitable for localhost/pilot use). Set API_KEY in .env to require an X-API-Key header on all requests. Set the key in the dashboard's ⚙ settings popover so the UI can authenticate.

GET /api/status

Current solar driver values, global risk level, and feed health. Polled by the dashboard every 60 s.

Response 200: { "global_risk_level": "NOMINAL", // NOMINAL | ELEVATED | DEGRADED | SEVERE "solar_drivers": { "kp_current": 2.0, "bz_nt": 0.0, "xray_class": "C", "solar_wind_km_s": 400.0, "proton_flux_10mev_pfu": 0.1 }, "data_age_seconds": 65, "feed_status": { "kp_index": "ok", "bz_gsm_nt": "ok" } }

POST /api/v2/route-decision

Per-waypoint and overall route risk decision for a platform.

Body: { "waypoints": [{ "lat": float, "lon": float, "name": str }], "platform": { "asset_type": "GPS_L1" | "GPS_L1L2" | "GPS_L1L5" | "GPS_INS" | "SBAS", "criticality": 1..5, "system_dependencies": [] } } Response 200 → action, action_sentence, alternatives, recommended_actions, confidence (score, label, drivers), impacts, waypoints, provenance

GET /api/v2/comms-decision

HF/SATCOM communications recommendation for a location (optional destination).

GET /api/v2/comms-decision?lat=38.8&lon=-77.0[&dest_lat=51.5&dest_lon=-0.1] Response → action (USE_PRIMARY_HF | SWITCH_TO_SATCOM | SWITCH_TO_UHF | HF_NOT_VIABLE | ...)

GET /api/forecast

72-hour Kp forecast windows with GPS/HF impact summaries per window.

GET /api/v2/replay

Replay a past decision from an archived NOAA snapshot. Use snapshot_id or at=ISO8601 to select the snapshot.

GET /api/v2/snapshots

List archived NOAA snapshots (paginated, most recent first). Response: {"snapshots": [...], "total": int}.

3D Globe Controls

  • Rotate: Left-click + drag
  • Zoom: Scroll wheel or right-click + drag
  • Tilt: Middle-click + drag (or Ctrl + drag)
  • Place waypoint: Click "⊕ Click map to add waypoint" then left-click the globe
  • Fly to route: Click the ⊙ button after adding waypoints

Building a Route

  1. Click "⊕ Click map to add waypoint" — cursor becomes a crosshair
  2. Click anywhere on the 3D globe to place a waypoint (auto-named WP01, WP02…)
  3. Or type lat/lon coordinates in the form and click "+ Add"
  4. Select your platform type from the dropdown
  5. Click "▶ Get Route Decision"

Reading a Decision

FieldMeaning
GOConditions are favorable. Proceed with standard monitoring.

Data Layers

Toggle layers in the top section of the right panel:

  • Ionosphere TEC — Auroral oval, equatorial anomaly, mid-latitude disturbance. Scales with live Kp.
  • Polar Cap Absorption — Active when proton flux triggers HF blackout. Red fill at high latitudes.
  • Solar Wind Alert — Live speed label. Green <400, Yellow <500, Orange <700, Red ≥700 km/s.

Forecast Timeline

The KP FORECAST bar at the bottom of the globe shows 72-hour forecast windows as a bar chart. Click any window to scrub the TEC layer to that forecast Kp. Click NOW to return to live conditions.

Frequently Asked Questions

How does IonShield get its data? +
IonShield polls NOAA Space Weather Prediction Center (SWPC) public feeds every 5 minutes: 1-minute Kp index, solar X-ray flux (GOES satellite), IMF Bz and solar wind speed from the ACE/DSCOVR L1 observatory, proton flux (≥10 MeV), and the NOAA 3-day Kp forecast. All data is archived for replay. Feed health is visible in the dashboard header.
Is IonShield certified for aircraft use? +
No. IonShield is a decision-support tool, not a certified avionics system. Outputs should be used to inform planning and raise situational awareness, not as the sole basis for navigation or safety-of-life decisions. Always verify against current NOAA SWPC advisories and applicable aviation weather briefings before flight.
How accurate are the GPS error estimates? +
GPS error estimates are derived from a physics-based ionospheric model parameterized by live Kp, Bz, and X-ray flux. They are indicative (within ±30–50% of observed values) rather than precision forecasts. For L1-only receivers, errors are higher; L1L2 dual-frequency receivers largely remove the ionospheric delay. The confidence score reflects data quality and model uncertainty.
Can IonShield operate in a disconnected / air-gapped environment? +
Yes, with caveats. IonShield can be packaged as a Docker container or Zarf/UDS bundle for disconnected deployment. Without live NOAA feeds, it operates in replay mode using the most recent archived snapshot. The dashboard's stale-data banner will activate, and confidence scores will be penalized. Contact us for air-gap deployment guidance.
Is the API ITAR-controlled? +
IonShield uses exclusively publicly available NOAA data (public domain) and open-source models. The software itself is assessed as EAR99 (no export license required under the EAR). It does not incorporate any restricted encryption, classified data, or controlled technical data. Please consult your organization's export-compliance officer before deploying in specific international contexts.
What is "deterministic replay" and why does it matter? +
Every NOAA observation archived by IonShield is stored with a SHA-256 input hash. The GET /api/v2/replay endpoint reconstructs the exact environment snapshot for any past timestamp and re-runs the decision engine — producing the same recommendation that would have been generated at that time. This enables after-action review, operator dispute resolution, model validation, and regulatory audit trails.
